CVE-2021-43798 Grafana 任意文件读取漏洞分析

发表于 2021-12-15 更新于 2021-12-18 分类于 漏洞分析

起因是因为打南航的ctf比赛,第一道web题目就是一个grafana的题目,经过搜寻直接利用新爆出来的任意文件读取漏洞,读到flag.所以复盘总结的时候来看一下这个漏洞.

漏洞描述

未授权用户可以任意读取grafana下的文件,但是不能直接getshell,需要利用读文件去配合其他漏洞,综合利用.

漏洞分析

通过阅读大佬们的文章,大体可以分为两种分析方式,1是从历史漏洞分析来推该漏洞2是从官方更新后的文件diff来看修改了那里.
这次主要是通过第一种,也是在如何网上出消息可以分析出来具体问题的方法(当然是看的别人的方法).
首先就是借鉴历史漏洞的思路,看到CVE-2020-13379这个经典漏洞,是从api.go文件入手,那我们也从这个文件入手.
首先从githib上下载grafan8.3.0的源码文件,然后在/pkg/api/api.go下查看代码.

可以看到很多个路由,第二个参数带有sign看起来像是是否需要验证.当然也有一些只有两个参数,那么我们就猜测,两个参数的就是不需要验证的,既然是未授权访问,那么就找只有两个参数的未授权的接口:


然后我们根据burp来验证一下具体的现象


所以我们可以根据注释来判断功能,看参数个数来判断是否授权,然后就需要找可以输入filepath的功能接口的函数,通过这种方式快速的过一下,找到这个plugins

跳转到getPluginAssets这个函数

根据整个函数的注释可以知道这个getPluginAssets是为了返回插件的公共资源,而他传递的格式为/public/plugins/插件名称/需要访问的资源
现在我们来大概分析一下代码
首先可以看到在277行处,插件的名称传递给了参数pluginID然后在278行处利用函数判断是否有这个插件,没有的话就会在258和260行处报404的错误.
284行处接受了filepath并且利用Clean函数清除了多余的字符,从*代表的路径和插件目录在285行处合并,但是没有过滤.
后续可以看到在获取到文件路径以后,没有过多的处理,295行处打开这个路径的文件,然后在322行处返回内容.
这个任意文件读取的漏洞就产生了.
然后找到一个内置的插件就可以使用这个漏洞了.
该漏洞后续的修复主要是利用一行
rel, err := filepath.Rel("/", requestedFile)
func Rel(basepath, targpath string) (string, error)
该函数以basepath为基准，返回targpath相对于basepath的相对路径，也就是说如果basepath为/a，targpath为/a/b/c，那么则会返回/b/c，如果两个参数有一个为绝对路径，一个为相对路径，则会返回错误
所以此时输入../则会无效化,无法路径穿越了

模版引擎

在jinja2中，存在三种语：

jinja2模板中使用 {{ }}语法表示一个变量，它是一种特殊的占位符。当利用jinja2进行渲染的时候，它会把这些特殊的占位符进行填充/替换，jinja2支持python中所有的Python数据类型比如列表、字段、对象等

python的ssti

当发现某个地方有ssti的时候，一般有两种思路

  • 读取配置文件
  • 执行系统命令

读取配置

pythonweb框架，比如说flask，在框架中内置了一些全局变量，对象，函数等等。我们可以直接访问或是调用。有的ctf会把flag放进config中，或者将敏感信息放到全局配置中，比如我们可以通过{{config}}读取配置文件

执行命令

Flask SSTI 题的基本思路就是利用 python 中的 魔术方法 找到自己要用的函数。

  • __dict__：保存类实例或对象实例的属性变量键值对字典
  • __class__：返回调用的参数类型
  • __mro__：返回一个包含对象所继承的基类元组，方法在解析时按照元组的顺序解析。
  • __bases__：返回类型列表
  • __subclasses__：返回object的子类
  • __init__：类的初始化方法
  • __globals__：函数会以字典类型返回当前位置的全部全局变量 与 func_globals 等价

base 和 mro 都是用来寻找基类的。主要是为了寻找object类

基本流程是：

  • 随便找一个内置类对象用__class__拿到他所对应的类
  • 用__bases__拿到基类（）
  • 用__subclasses__()拿到子类列表
  • 在子类列表中直接寻找可以利用的类getshell。

解决蚁剑执行mssql数据库命令执行报错问题

发表于 2021-12-08 更新于 2021-12-09 分类于 渗透测试

因为公司的需求要去搞渗透赛,里面正好有一个场景就是mssql的数据库的命令执行然后获取flag,但是按照wp执行以后产生了问题,出现了**-2147217887:ODBC 驱动程序不支持所需的属性**

此时就产生了怀疑,以为是网络或者是蚁剑的版本的问题,因为在靶场平台是可以执行的,于是将蚁剑切换了版本,还是不行.
本着搜索引擎的强大,直接在百度搜索一只出现答非所问的问题,最终在google发现了解决的办法.

原因是在source\core\asp\template\database\sqlserver.js中执行SQL语句代码模板中使用Recordset对象的open方法来执行SQL语句，貌似是该方法不支持请求xp_cmdshell这种过程对象，具体我也不清楚，这里参考了下菜刀的方式解决。
删除102、103行代码

更改为Connection对象的Execute方法直接执行
Set Rs=Conn.Execute(""&bd(Request("${arg2}"))&""):
为了防止执行一些返回recordset.Fields为0的命令（如开启xp_cmdshell的命令：EXEC sp_configure “show advanced options”,1）后面程序进入死循环的情况，这里还需要加一个if判断：
在FN=Rs.Fields.Count-1:的后面添加
If FN=-1 Then: Response.Write HD&"Execute Successfully!"&CO&RN: Else:
再在loop后面添加End If:

最后测试成功.
当然也可以利用navcat数据库连接然后去执行命令,或者利用菜刀2014都是可以的.
